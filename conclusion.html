Rationale for Web Bluetooth




WebBluetooth API on browser is an incredibly nice feature, which will allow the user to interact without any pain point to install drivers or software. Their present limitation makes the development to think on its scope. As of now only Chrome browser supports this feature that too only on Linux it works as standalone, on Windows it needs third party plugin.


In order to communicate with a device, that device needs to support a GATT Service that the web page understands and has permission to access. If the web page doesn't filter the devices that appear in the requestDevice() dialog, users could easily select a device the web page can't use, which is a bad experience.


requestDevice() will give list of device and hooking on the running services is difficult since that requires authentication. We tried it with real time device (Smart band, Glucometer, Fitness tracker) the experiment was not as we expected. Sure that one can access public services available for the device to get information, but to get more information on the device like tracking, it is not possible unless proper authentication is made. This is a pain point and hope in future release Google might address the same. Considering the limitations and leverages one can use this WEB BLUETOOTH API feature.
